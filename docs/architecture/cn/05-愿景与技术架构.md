# 愿景与技术架构

### 📄 项目核心文档：SynapticSYS 平台愿景与技术架构

​**文档状态**​：内部讨论稿  
​**整理日期**​：2023年10月27日  
​**核心目标**：定义"SynapticSYS"——一个基于ROS2构建的、面向高复杂度实时系统的全链路、图形化集成开发与运行时平台。

---

### 第一部分：核心愿景与需求总览

**1.1 项目定位**  
SynapticSYS旨在成为工业自动化、机器人、数字孪生等领域中，用于构建**高复杂度实时系统**的下一代开发平台。它将传统SCADA、组态软件、现代分布式架构与可视化开发工具深度融合。

**1.2 核心需求摘要**  
您提出的需求远超单一工具，是一个完整的生态系统：

- ​**架构**​：基于ROS2的​**多进程、多节点分布式微服务架构**​，原生支持 **C++、C#、Python** 混合编程。
- ​**开发体验**​：提供一站式图形化IDE，集成​**蓝图编辑器**​（逻辑流）与​**UI编辑器**（HMI）。
- ​**运维与调试**​：具备​**全链路可观测性**，包括日志回放、HMI录屏、远程调试与桌面。
- ​**生产部署**​：支持从开发到生产的一站式​**构建、打包、容器化部署**。
- ​**生态与兼容**：基于ROS2生态系统，充分利用其成熟的工具链和包管理机制。
- ​**高级特性**​：探索**同一进程内运行跨语言模块**的机制，以灵活平衡调试效率与运行隔离性。

**1.3 已确定的关键决策**

- ​**命名**​：项目主品牌名为 ​**SynapticSYS**​。其构建系统可称为 `synaptic-build`​，命令行工具为 `synaptic`。
- ​**开源协议**​：采用 ​**Apache License 2.0**，兼顾商业友好性与明确专利授权。
- ​**项目结构**​：平台代码（`synaptic-platform`​）与用户项目代码必须​**物理分离**​。平台自身采用 **Monorepo** 单一代码库管理所有组件。
- ​**代码目录**​：源码根目录统一使用 ​**​`src`​**。
- ​**协作规范**​：采用 `Conventional Commits`​ 规范，文档化于 `CONTRIBUTING.md`。
- ​**组织与仓库**​：在Gitee/GitHub创建统一组织（如 `synapticsys`​），平台主仓库命名为 `platform`。

### 第二部分：核心技术栈与架构决策

**2.1 构建系统与包管理器**

- ​**结论**​：基于ROS2的Colcon/Ament构建系统，扩展支持C++、C#、Python、JavaScript多语言开发。
- ​**核心理念**：

  - ​**基于Colcon的"协调者"架构**：利用ROS2成熟的元构建工具，负责包发现、依赖解析、任务调度。
  - ​**扩展支持多语言**：通过插件机制扩展Colcon，支持C++、C#、Python、JavaScript混合编程。
  - ​**以** **​`package.xml`**​ **和** **`synaptic.yaml`​**​ **为配置源**：结合ROS2标准的package.xml和扩展的synaptic.yaml，定义项目与包的元数据、依赖。
- ​**技术栈**​：基于Colcon/Ament构建系统，通过插件调用各语言原生工具：
  - **C++**：`CMake`/`ament_cmake`
  - **C#**：`dotnet`/`MSBuild`
  - **Python**：`pip`/`setup.py`
  - **JavaScript/TypeScript**：`npm`/`yarn`/`webpack`

- ​**多语言支持实现**：
  - **C++**：直接使用ROS2的`ament_cmake`构建系统
  - **Python**：使用ROS2的`ament_python`构建系统
  - **C#**：开发`ament_dotnet`插件，集成dotnet构建系统
  - **JavaScript/TypeScript**：开发`ament_js`插件，集成npm/yarn构建系统

- ​**包结构规范**：
  - **C++包**：遵循ROS2 C++包结构，使用`CMakeLists.txt`和`package.xml`
  - **Python包**：遵循ROS2 Python包结构，使用`setup.py`和`package.xml`
  - **C#包**：扩展ROS2包结构，使用`*.csproj`和`package.xml`
  - **JavaScript包**：扩展ROS2包结构，使用`package.json`和`package.xml`

**2.2 通信与运行时**

- ​**基于ROS2通信系统**​：利用ROS2成熟的DDS通信机制，包括话题、服务、动作等，作为系统的核心通信基础。
- ​**通信抽象层**​：设计 **​`Synaptic Link`​** （通信管理器），在ROS2通信系统基础上扩展，以插件化形式支持DDS、MQTT、WebSocket等多种协议。
- ​**跨语言通信方案**：
  - **C++ ↔ C#**：使用.NET Core的P/Invoke或C++/CLI实现
  - **C++ ↔ JavaScript**：使用Emscripten或Node.js原生模块实现
  - **Python ↔ C#**：使用Python.NET库实现
  - **Python ↔ JavaScript**：使用PyExecJS或类似库实现
  - **C# ↔ JavaScript**：使用Blazor或Node.js的Edge.js实现

- ​**运行时服务管理**：基于ROS2节点生命周期管理，实现多语言服务管理器，负责微服务节点的生命周期（启动、停止、监控、依赖）。
- ​**依赖管理**​：结合ROS2的package.xml和扩展的synaptic.yaml，明确定义 `build_deps`​（构建时依赖）和 `exec_deps`（运行时依赖），由构建工具解析并确保传递。
- ​**多语言运行时支持**：
  - **C++**：直接使用ROS2的rclcpp
  - **Python**：使用ROS2的rclpy
  - **C#**：开发rclcs（ROS2 C#客户端库）
  - **JavaScript/TypeScript**：开发rcljs（ROS2 JavaScript客户端库）

**2.3 跨语言进程内微服务机制**

- ​**结论**​：这是定义平台先进性的​**核心高级特性**，但建议分阶段实现，扩展到支持JavaScript。
- ​**参考项目**​：可深度借鉴 ​**CppMicroServices**​、​**CTK**​、**Apache Celix** 等基于OSGi理念的框架，学习其**动态库（Bundle）加载、服务注册与发现、生命周期管理**的机制。
- ​**实现路径**：

  1. 设计语言无关的​**统一服务抽象层**，支持C++、C#、Python、JavaScript。
  2. 为C++、C#、Python、JavaScript分别实现​**运行时适配层**，处理与统一层的桥接、内存管理和垃圾回收协调。
  3. 实现​**配置驱动**，允许服务在“进程内插件”与“独立进程”两种模式间切换。

- ​**JavaScript支持实现**：
  - **Node.js集成**：使用Node.js的原生模块机制
  - **WebAssembly集成**：使用WebAssembly实现跨平台的JavaScript执行
  - **V8引擎嵌入**：直接在C++中嵌入V8引擎执行JavaScript代码

- ​**多语言统一抽象层**：
  - **服务注册与发现**：基于ROS2服务发现机制的多语言服务注册与发现
  - **生命周期管理**：统一的服务生命周期管理，支持四种语言的服务
  - **数据交换格式**：使用ROS2消息格式作为跨语言数据交换的标准格式

**2.4 容器化与生态兼容**

- ​**结论**​：基于ROS2生态系统，充分利用其成熟的容器化支持和生态兼容能力，扩展支持多语言。
- ​**方案**：直接基于ROS2构建，利用其原生的容器化支持和生态系统。通过ROS2的标准接口，实现与ROS2生态系统的无缝集成，同时保持系统架构的简洁性和一致性。

- ​**多语言容器化方案**：
  - **C++/Python容器**：直接使用ROS2官方容器镜像
  - **C#容器**：基于ROS2镜像，添加.NET Core运行时
  - **JavaScript容器**：基于ROS2镜像，添加Node.js运行时
  - **多语言混合容器**：包含所有语言运行时的统一容器镜像

- ​**容器镜像优化**：
  - **分层构建**：使用Docker多阶段构建减少镜像大小
  - **运行时镜像**：为生产环境提供最小化运行时镜像
  - **开发镜像**：为开发环境提供完整工具链

- ​**生态系统集成**：
  - **ROS2包集成**：直接使用现有的ROS2包
  - **C#生态集成**：访问.NET生态系统中的库
  - **JavaScript生态集成**：访问npm包生态系统
  - **跨生态系统桥接**：实现不同生态系统间的无缝集成

**2.5 IDE集成策略**

- ​**结论**​：优先为 **VS Code** 提供一流的开发体验，作为差异化优势，扩展支持JavaScript开发。
- ​**方案**​：`synaptic ide-integrate` 命令自动生成：

  - ​ **​`.vscode/tasks.json`​**：为所有包生成构建、测试任务，包括C++、C#、Python、JavaScript。
  - ​ **​`.vscode/launch.json`​**：为可执行服务生成调试配置，支持四种语言。
  - ​**​`compile_commands.json`​**：为C++代码提供完美的智能感知。
  - **`.vscode/settings.json`**：为JavaScript/TypeScript提供智能感知配置。

- ​**多语言IDE支持**：
  - **C++**：利用VS Code的C/C++扩展
  - **Python**：利用VS Code的Python扩展
  - **C#**：利用VS Code的C#扩展
  - **JavaScript/TypeScript**：利用VS Code的ESLint、Prettier等扩展

- ​**统一调试体验**：
  - **跨语言调试**：实现C++、C#、Python、JavaScript的统一调试体验
  - **断点同步**：在多语言调用链中保持断点同步
  - **变量查看**：统一查看不同语言中的变量

- ​**注**：不推荐生成一个扭曲的、包含所有语言项目的“万能CMake文件”，这会导致构建逻辑混乱和IDE功能降级。

### 第三部分：关键挑战与未决事项

**3.1 待解决的核心技术挑战**

1. ​**跨语言数据交换与内存管理**：在同一进程内，安全高效地在C++、C#、Python、JavaScript间传递复杂数据是最大挑战，需设计精密的所有权与生命周期协议。
2. ​**多语言统一调试体验**：如何让开发者在IDE中无缝地调试一个跨越四种语言服务调用链，需要工具链的深度定制。
3. ​**性能与资源权衡**：进程内模式与独立进程模式在延迟、资源隔离和可靠性之间的具体权衡指标需通过原型实测确定。
4. ​**JavaScript执行效率**：在实时系统中，JavaScript的执行效率可能成为瓶颈，需要探索优化方案。
5. ​**垃圾回收协调**：四种语言的垃圾回收机制不同，需要设计协调机制避免内存泄漏。

**3.2 近期行动建议**

1. **基于ROS2构建最小原型**：使用ROS2的Colcon/Ament构建系统，实现支持C++、Python混合编程的最小原型。
2. **扩展Colcon支持多语言**：开发Colcon插件，扩展支持C#和JavaScript等其他语言。
3. **验证跨语言通信原型**：验证C++、C#、Python、JavaScript通过ROS2通信机制的跨语言通信可行性。
4. **开发多语言客户端库**：
   - 开发rclcs（ROS2 C#客户端库）
   - 开发rcljs（ROS2 JavaScript客户端库）
5. **实现JavaScript集成**：探索Node.js集成和WebAssembly方案，验证JavaScript在实时系统中的性能。
